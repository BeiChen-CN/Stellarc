# Git

## 分支与提交策略

- 默认在当前分支顺序修改；未要求不创建分支/worktree。
- 已有未提交改动时先收口再继续，禁止堆叠新任务。
- 验证后用户要求时提交，不 amend 旧 commit。
- 未明确要求不执行 push/rebase/reset/amend/force push/删分支/清 worktree。

## Commit 规范

- 格式 `<type>(scope): <summary>`；scope 可选；summary 中文、动词开头、≤50 字、无句号。
- 常用 type：feat/fix/refactor/docs/test/chore。优先使用项目既有风格。

## 去除 AI 痕迹

- commit/PR/源码/注释/文档中禁止 AI 署名、工具痕迹（Generated by...、Co-Authored-By:...、AI-generated、🤖、Agent、LLM、prompt）及过程描述。
- PR 描述只写：问题背景、改动内容、验证方式、风险与回滚。
- 不混入格式化/重排/命名风格变化到功能提交；不写中间推理/试错/候选方案。
- 避免 AI 式夸张表达和冗长总结。
- 保留真实作者、许可证、版权声明；遵守披露要求。
- 提交前检查 diff/message/PR/分支名/新文件，确认无 AI 痕迹/调试残留/临时注释。
- 整理历史/合并 commit/改写 message/rebase 须用户确认。
